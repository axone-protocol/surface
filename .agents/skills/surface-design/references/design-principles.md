# Design Principles

These principles govern every design decision made in Surface.

Component specifications, layouts and visual patterns must derive from them.

## Start from the protocol

Surface represents the Axone protocol.

Every interface begins by identifying the protocol primitive it exposes.

## Design institutional consequences

Users interact with institutional facts, not protocol mechanics.

Interfaces express what becomes true once a protocol primitive has been recorded.

## Reveal, don't interpret

Display only what can be established from protocol facts.

Do not infer additional meaning.

Derived concepts belong to higher-level views.

## Separate assertion from evidence

Every institutional fact has two representations:

- the assertion;
- the evidence supporting it.

Assertions communicate.

Evidence verifies.

Never mix both concerns.

## Prefer stable references

Institutional facts should be identifiable, addressable and inspectable.

References are first-class interface elements.

## Progressive inspection

Present only the information required to understand an institutional fact.

Expose technical details progressively through inspection.

## Motion expresses state

Animation communicates protocol activity.

Motion must stop when the represented state becomes stable.

## Visual meaning

Every visual decision communicates structure.

Typography, spacing, hierarchy, color and timing are semantic elements.

Decoration without meaning does not belong in Surface.

## Consistency over originality

Every new interface should extend the existing visual language.

Create new patterns only when existing ones cannot express the required concept.

## Build inevitable interfaces

Do not imitate existing blockchain explorers.

Do not imitate administrative documents.

Do not imitate SaaS dashboards.

Design the interface that a native digital institution would naturally produce.
